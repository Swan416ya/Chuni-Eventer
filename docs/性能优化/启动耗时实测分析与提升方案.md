# 启动耗时实测分析与提升方案

本文基于一次真实冷启动的 `[startup-profile]` 输出（`python -m chuni_eventer_desktop`，Windows，`.venv-build`），总耗时约 **5.8s** 到首屏可交互。数据来自 `.cache/logs/startup_profile.log` 与同次终端摘要。

---

## 1. 结论（先看这个）

| 阶段 | 累计 | 单段耗时 | 占总量 |
|------|------|----------|--------|
| 窗口首次 `show()` 完成 | **5746ms** | — | **99%** 用户体感「还没出来」 |
| 歌曲列表可滚动 | **5781ms** | +35ms | 几乎与 show 同时 |
| 进程启动 → 进入事件循环 | 5751ms | — | — |

**真正拖慢启动的不是歌曲 scan / 封面 / 游戏索引**（合计 <50ms），而是：

1. **MainWindow 构造阶段约 4.8s**（其中有一段 **3.7s 未命名空档**）
2. **SettingsPage 同步构造 ~613ms**
3. **首次 `show()` 绘制 ~400ms**
4. **Python 冷 import ~464ms**（274 + 190）

优化应优先：**缩短「show 之前」的主窗口构造**，而不是继续改 Music 卡片或索引逻辑。

---

## 2. 时间线（按发生顺序）

```
T+0ms      进程启动
T+40ms     startup_begin
T+314ms    import app 完成（PyQt / qfluentwidgets / 业务模块）
T+367ms    QApplication 就绪
T+571ms    main_window 模块 import 完成（顶层 import 全部 *Dialog）
T+784ms    MSFluentWindow.super().__init__ 完成
           │
           │  ⚠ 空档 3700ms — 见第 3 节
           ▼
T+4485ms   开始读 config（AcusConfig + ensure_acus_layout）
T+4496ms   config 读完（仅 11ms，不是瓶颈）
T+4607ms   ManagerWidget 构造完成（~70ms）
T+5262ms   SettingsPage 构造完成（~613ms）
T+5338ms   MainWindow.__init__ 结束
T+5347ms   调用 show() 之前
T+5746ms   show() 返回（首次绘制 ~400ms）
T+5781ms   默认「歌曲」reload 完成（scan 8 首 + 卡片 ~20ms）
T+5807ms   游戏索引 cache 命中（25ms），封面后台 8 张 ~25ms
```

---

## 3. Top 瓶颈详解

### 3.1 ⚠ 空档 3700ms：`super().__init__` 之后、`config` 之前

日志片段：

```
[   784.4ms] MainWindow.super().__init__:end
[  4484.8ms] MainWindow.config:begin     ← +3700.5ms
[  4496.3ms] MainWindow.config:end       ← 仅 +11.5ms
```

这两行 mark 之间，源码**只有**：

```python
self.setWindowTitle(...)
self.resize(1160, 824)
```

因此这 3.7s **不是**读配置、不是扫 ACUS、不是扫游戏盘，而是：

- **Fluent `MSFluentWindow` 在首次 `resize` / 标题栏初始化时触发的布局与渲染准备**（导航栏、无边框窗口、DWM 合成等），或
- 杀软 / 同步盘对首次窗口操作的放大延迟（需在同一机器上用更细 mark 验证）

`ensure_acus_layout` 已轻量化，**不要再从这里挖**；旧文档里 16s works sort 问题在你这次日志里已不存在。

**建议下一步埋点**（实现优化前先做）：

```text
MainWindow.after_setWindowTitle
MainWindow.after_resize
MainWindow.config:begin
```

以确认是 `resize` 还是 `setWindowTitle` 触发 3.7s。

---

### 3.2 SettingsPage 同步构造 ~613ms

```
[  4649.2ms] MainWindow.SettingsPage:begin
[  5261.8ms] MainWindow.SettingsPage:end   ← +612.5ms
```

`SettingsPage` 在 `MainWindow.__init__` 里**一次性**创建五个子面板（关于 / 游戏数据 / 外部工具 / 存档 / 实验性）。用户启动后默认看「歌曲」，**设置页完全可以懒加载**。

**预期收益**：启动路径减少 **~600ms**（你这台机器上的实测值）。

---

### 3.3 首次 `show()` ~400ms

```
[  5346.8ms] app.main:before_show
[  5746.3ms] app.main:after_show          ← +399.5ms
```

窗口在 **已堆满 ManagerWidget + SettingsPage + 导航** 之后才 show，首次 paint 要合成大量控件。

**预期收益**：若改为「轻壳先 show、重 UI 延后」，这 400ms 可能部分与构造重叠，且用户 **T+1~2s 就能看到窗口**，体感远好于 T+5.7s。

---

### 3.4 Python 冷 import ~464ms

| 段落 | 耗时 |
|------|------|
| `__main__:after_app_import` | 274ms |
| `main_window:module_imported` | 190ms |

`main_window.py` 顶层 import 了十余个 `*AddDialog`、下载对话框等，启动即加载。

**预期收益**：按需 import 约 **150~250ms**（视机器而定）。

---

### 3.5 次要项（本次可忽略）

| 项目 | 耗时 | 说明 |
|------|------|------|
| `MSFluentWindow.super().__init__` | 211ms | 基类构造，难完全去掉 |
| `ManagerWidget.__init__` | 70ms | 可接受 |
| `scan_music` + 卡片 | ~20ms | 8 首歌 |
| `load_cached_game_index` | 25ms | 已有缓存 |
| 封面 `_JacketLoadWorker` | 25ms | 已后台化 |

## 4. 耗时占比（show 之前）

```
show 之前合计 ≈ 5347ms

  空档 setWindowTitle/resize 后   ~3700ms   69%  ← 最大头
  SettingsPage                    ~613ms    11%
  首次 show 前已完成的 Fluent/UI   ~400ms    8%  （含 super 211 + import 190 等）
  ManagerWidget + 其它 UI         ~150ms    3%
  config / ACUS 目录               ~12ms    <1%
  Python import（show 前部分）     ~464ms    9%
```

（百分比为近似，部分阶段并行感不强，但优先级清晰。）

---

## 5. 提升方案（按优先级）

### 阶段 A — 立刻可做，收益最大（目标：TTFW 2~3s）

#### A1. 设置页懒加载（P0，~600ms）

- **做法**：`SettingsPage` 改为首次点击「设置」导航时再 `new SettingsPage(...)`，或 `QStackedWidget` + factory。
- **改动**：`ui/settings_page.py`、`ui/main_window.py`（`_enter_settings_mode`）。
- **风险**：低；首次进设置多等 ~600ms 可接受。

#### A2. 主窗口「轻壳先 show」（P0，体感最大）

- **做法**：
  1. `super().__init__()` + 读 config + `ensure_acus_layout`（~15ms）
  2. **立即 `show()`**（空 workspace 或 loading 占位）
  3. `QTimer.singleShot(0, _build_heavy_ui)` 里再建 ManagerWidget、导航、Settings 占位
- **目的**：把 3.7s 空档 + 613ms Settings + 部分 400ms show 成本移到 **窗口已可见之后**。
- **预期**：用户 **~0.8~1.5s 看到窗口**，5.8s 总工作量不变但「可交互感」提前很多。

#### A3. 细分并针对性处理 3.7s 空档（P0）

在确认 `resize` 是主因后，可尝试（择一或组合）：

- 推迟 `resize` 到 show 之后；
- 缩小首次 show 尺寸再 `showMaximized`；
- 检查 `MSFluentWindow` / 导航是否在 `resize` 时同步构建大量 SVG 图标（`nav_qicon` 等）— 可改为懒加载图标。

---

### 阶段 B — 中优先级（再省 ~200~400ms 固定成本）

#### B1. `main_window.py` 对话框按需 import

- `_on_add`、各入口第一次打开时再 `import TrophyAddDialog` 等。
- 预计减少 `main_window:module_imported` 大部分 190ms。

#### B2. 延迟 import 非首屏模块链

- `app.py` 中 `MainWindow` 已在 `main()` 内 import，可保持；
- 考虑 `manager_widget` 对 `CharaAddDialog` 等改为 lazy。

#### B3. 导航图标懒加载

- `nav_qicon(SVG_*)` 若在 import/resize 时 rasterize，可缓存到磁盘或首次 hover 再加载。

---

### 阶段 C — 长期 / 可选

| 项 | 说明 |
|----|------|
| ACUS 分类 JSON 缓存 | 切换 Tab 更快，对冷启动帮助有限 |
| 打包 onefile 优化 | 若用 exe 分发，另有一轮解压成本 |
| 启动 Splash | 在 A2 未做前过渡方案；不如轻壳 show 干净 |

---

## 6. 推荐实施顺序

| 顺序 | 任务 | 预期（你的机器） | 工作量 |
|------|------|------------------|--------|
| 1 | SettingsPage 懒加载 | −600ms 构造 | 小 |
| 2 | 轻壳先 show + 延后 heavy UI | 体感 TTFW 从 5.7s → ~1~2s | 中 |
| 3 | 细分 resize 空档 + 针对性 fix | −部分 3700ms | 小~中 |
| 4 | main_window 按需 import | −~190ms | 小 |
| 5 | 验证后关闭/保留 startup_profile | — |  trivial |

**不建议再投入**：歌曲封面同步化、启动时全量游戏索引（你这次 cache 命中且仅 25ms）。

---

## 7. 验收标准

| 指标 | 当前（实测） | 阶段 A 目标 |
|------|--------------|-------------|
| TTFW（窗口可见） | **5746ms** | **≤ 2000ms** |
| 歌曲列表可操作 | 5781ms | ≤ 2500ms（可与 TTFW 接近） |
| `MainWindow.__init__` 阻塞 show 前 | 5338ms | **< 500ms**（重 UI 移出） |
| SettingsPage 占用启动 | 613ms | **0ms**（懒加载） |

每轮优化后对比 `.cache/logs/startup_profile.log` 中 **3s-final 摘要** 前 5 项 delta。

---

## 8. 如何复测

```powershell
# 默认已开启；关闭：$env:CHUNI_STARTUP_PROFILE='0'
python -m chuni_eventer_desktop
```

关注：

1. `app.main:after_show` 累计 ms（TTFW）
2. `MainWindow.config:begin` 的 **+delta** 是否仍 ~3700ms
3. 是否还有 `MainWindow.SettingsPage:end` 大 delta（懒加载后应消失于启动路径）
