# 设置页读取游戏根目录资源链路梳理

本文聚焦「设置」中 `游戏数据目录(game_root)` 相关逻辑，目标是给后续修复**读取卡死**与**闪退**提供一份可执行的链路图与风险清单。

## 1. 入口与配置持久化

### 1.1 设置页入口
- 入口：`chuni_eventer_desktop/ui/main_window.py` 的 `_open_settings()`
- 弹窗：`chuni_eventer_desktop/ui/settings_dialog.py` 的 `SettingsDialog`
- 关键字段：
  - `self.game_root`（游戏目录）
  - `self.compressonator`（DDS 工具路径，索引时用于可选预热）

### 1.2 保存行为
- `SettingsDialog.apply()` 将 `game_root` 写入 `AcusConfig.game_root`
- `AcusConfig.save()` 落盘到：`ACUS/.config.json`

### 1.3 触发重扫行为
有两条触发链：
- 在设置页点击「重新扫描游戏索引」：
  - `SettingsDialog._rescan_game_index()` 调用传入回调 `on_request_game_rescan(root, tool_path)`
- 点击设置页「保存」并关闭：
  - `MainWindow._open_settings()` 中 `dlg.apply()` 后，如果 `game_root` 非空也会调用 `_request_game_index_rescan(...)`

结论：用户在设置页一次操作可能触发重扫；但主窗口有并发保护（见 2.2）。

---

## 2. 后台扫描执行模型（主窗口）

### 2.1 自动扫描（启动后）
- `MainWindow.__init__()` 里 `QTimer.singleShot(0, self._ensure_game_index_background)`
- 当 `game_root` 配置存在且无缓存时，自动 `self._start_index_thread(gr)`

### 2.2 手动重扫（设置触发）
- `MainWindow._request_game_index_rescan(...)`
- 如果已有扫描线程：直接提示“扫描进行中”并返回
- 没有线程才会进入 `_start_index_thread(...)`

### 2.3 线程与 UI 交互
- 工作线程对象：`_GameIndexWorker(QObject)`
- 实际调用：`rebuild_and_save_game_index(..., prewarm_dds_preview=False)`
- 进度通过 `progress` signal 推给主线程，更新 `QProgressDialog`
- 完成后：
  - 成功：提示索引数量，并执行 `refresh_chara_works_sorts_with_game(...)`
  - 失败：提示错误，指导用户去设置页重新选择目录

关键点：主索引构建在后台线程执行，理论上不应阻塞主界面事件循环。

---

## 3. 游戏根目录识别与扫描范围

核心在 `chuni_eventer_desktop/game_data_index.py`。

### 3.1 数据包根识别 `enumerate_game_data_roots(game_root)`
- 先 `game_root.expanduser().resolve()`，要求目录存在
- 判定目录是否像数据包根：含任一 marker 子目录：
  - `music`, `stage`, `ddsImage`, `ddsMap`, `chara`, `reward`, `releaseTag`
- 兼容用户直接选中某个包根（自身符合 marker）：
  - 会直接加入 roots
- 主规则：`root.rglob("*")` 递归全量遍历
  - 筛选目录名长度 4 且以 `A` 开头（即 `A???`）
  - 且该目录也需满足 marker 判断
- 最后 resolve + 去重输出

### 3.2 索引构建 `build_game_data_index(...)`
扫描流程：
1. 枚举数据包根 `roots`
2. 逐包调用 `scan_music(pack)`，合并出 `music_catalog` 与 `music`
3. 汇总 stage / ddsImage / ddsMap：
   - `_aggregate_stages_dds_from_roots()`
4. 汇总 chara / namePlate / trophy：
   - `_aggregate_chara_np_trophy_from_roots()`
5. 生成 `GameDataIndex` 对象

### 3.3 落盘缓存
- `save_game_data_index(idx)` 写入 `.cache/game_data_index.json`
- 加载缓存：`load_cached_game_index(expected_game_root=...)`
  - 若 `expected_game_root` 与缓存中 `game_root` resolve 后不一致，则缓存无效

## 4. 索引数据被谁读取（设置链路的下游）

索引输出并非只用于“浏览乐曲”，而是供多个编辑器下拉复用。

### 4.1 主读取入口
- `MainWindow._resolve_game_index()` -> `load_cached_game_index(self._cfg.game_root)`
- `ManagerWidget`、新增弹窗等通过回调拿到当前索引

### 4.2 常见消费点
- `MapAddDialog`：
  - `merged_music_pairs`
  - `merged_dds_map_pairs`
  - `merged_stage_pairs`
- `MapBonusEditDialog`：`merged_music_pairs`
- `PgkoSheetDownloadDialog`：`merged_stage_pairs`
- `GameMusicBrowserDialog`：
  - 优先读缓存 `idx.music_catalog`
  - 没缓存时会直接 `scan_game_music_catalog(self._game_root)` 读磁盘

### 4.3 合并策略
`merged_*_pairs(acus_root, idx)` 统一策略：
- 先放游戏索引数据
- 再覆盖 ACUS 工作区数据（同 ID 以 ACUS 为准）
- 输出排序后的 `(id, name)` 列表

## 5. 与“卡死 / 闪退”最相关的高风险点

下面仅列当前代码链路中最值得优先排查的点。

### 5.1 风险 A：`enumerate_game_data_roots()` 的全盘 `rglob("*")`
现象特征：
- 目录很大、存在软链接/junction、网络盘或慢盘时，扫描耗时可能极高
- 虽在后台线程，但线程长期占用会导致“看起来一直在扫”

风险级别：高（卡慢/看似卡死）

### 5.2 风险 B：`GameMusicBrowserDialog` 的前台直扫
触发条件：
- 打开“游戏乐曲资源（只读）”时若无缓存
- 或点击“从磁盘重新扫描”

行为：
- 直接在 UI 线程调用 `scan_game_music_catalog()`，内部会遍历所有包并解析大量 XML

风险级别：高（真实 UI 卡死感）

### 5.3 风险 C：路径 resolve 与 I/O 异常边界
多个位置使用 `Path.resolve()` / `is_dir()` / `glob()` / XML parse。
- 大多数异常已被 `except Exception/OSError` 吞掉并继续
- 但个别 UI 初始化路径（如浏览对话框构造时路径处理）一旦遇到非常规路径/权限问题，仍可能出现未覆盖异常

风险级别：中（偶发闪退）

### 5.4 风险 D：扫描结果体量与提示机制
- 当前 `QProgressDialog` 无取消按钮
- 用户若选错目录（例如磁盘根），只能等待任务结束
- 误判为“应用卡死”概率较高

风险级别：中高（体验型“卡死”）

---

## 6. 建议的修复优先级（用于下一步实现）

### P0（先做）
1. **限制扫描范围**  
   在 `enumerate_game_data_roots()` 避免无边界 `rglob("*")`，优先定向探测：
   - `data/*/opt/*`
   - `bin/option/*`
   - 用户当前目录本身是否为包根
2. **为后台扫描增加超时/取消能力**  
   允许取消线程任务，避免错误目录导致长时间不可控扫描。
3. **把 `GameMusicBrowserDialog` 的磁盘扫描改为后台线程**  
   与主索引同样使用 worker + progress，避免 UI 线程直扫。

### P1（随后）
1. 路径与权限异常统一兜底（集中封装安全文件系统访问）
2. 为索引缓存增加“最近成功扫描摘要”与“失败原因统计”
3. 对单包扫描耗时做日志埋点（哪个包最慢、哪类 XML 最慢）

---

## 7. 便于排障的最小调用链（速查）

1. 设置页保存/重扫  
`SettingsDialog` -> `MainWindow._request_game_index_rescan`

2. 后台线程扫描  
`MainWindow._start_index_thread` -> `_GameIndexWorker.run` -> `rebuild_and_save_game_index`

3. 资源枚举与索引构建  
`rebuild_and_save_game_index` -> `build_game_data_index` -> `enumerate_game_data_roots` + `scan_*`

4. 缓存读取与业务下拉  
`load_cached_game_index` -> `merged_*_pairs` -> 各新增/编辑弹窗

5. 一个例外（前台直扫）  
`GameMusicBrowserDialog._load_rows/_reload_from_disk` -> `scan_game_music_catalog`

---

## 8. 结论

设置页本身主要负责“配置 + 触发”，真正的资源读取重负载在 `game_data_index`。  
当前最可能造成“卡死体感”的核心不是单点崩溃，而是：
- 无边界目录遍历
- 个别对话框仍在 UI 线程做全盘扫描

后续修复建议先围绕这两条主线做结构性改造，再补异常边界与可观测性。
