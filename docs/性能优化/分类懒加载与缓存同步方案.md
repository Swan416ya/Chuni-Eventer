# 分类懒加载与缓存同步方案

## 目标

- 启动阶段只加载当前分类（默认歌曲）
- 其它分类在用户点击时再读取 XML
- 通过缓存减少重复 XML 解析
- 增删改后缓存与 `dds_preview` PNG 保持一致

## 当前状态（已满足/已完成）

- `ManagerWidget` 已按当前分类扫描，不会一次性扫描所有分类
- 启动默认进入 `Music`，其余分类点击后才 `reload()`
- 已将 `ensure_acus_layout` 中的同步 works sort 刷新移出启动主链路（避免 10s+ 阻塞）

## 建议架构

## 1) 分类缓存文件

- 缓存位置：`.cache/acus_index/`
- 每个分类一个文件，例如：
  - `music.json`
  - `map.json`
  - `event.json`
  - `stage.json`
  - `dds_image.json`
- 缓存内容：
  - `version`
  - `kind`
  - `built_at`
  - `fingerprint`
  - `items`（列表数据）

## 2) 指纹（fingerprint）策略

- 不做全量文件 hash（太慢）
- 使用目录级轻量签名：
  - `count`（XML 文件数）
  - `latest_mtime_ns`
  - `total_size`（可选）
- 按分类计算目标目录签名，例如：
  - `Music` -> `ACUS/music`
  - `Stage` -> `ACUS/stage`
  - `DDSImage` -> `ACUS/ddsImage`

## 3) 读取流程（点击分类时）

1. 计算当前分类 fingerprint
2. 命中缓存且 fingerprint 一致 -> 直接读缓存
3. 否则 -> 扫描 XML -> 回写该分类缓存

## 4) 增删改同步规则

- 所有“写 XML”的入口在成功后调用 `invalidate_kind_cache(kind)`
- 所有“删除目录/XML”的入口在成功后调用 `invalidate_kind_cache(kind)`
- 下一次进入该分类自动重建缓存

### 推荐先接入的操作点

- 新增/编辑/删除：`Music`、`Map`、`Event`、`Quest`、`Trophy`、`NamePlate`、`Stage`、`MapBonus`、`DDSImage`

## 5) DDS 预览 PNG 缓存同步

- 路径：`.cache/dds_preview/*.png`
- 同步规则：
  - 当源 `.dds` 被替换/删除时，删除对应 `.png`
  - 下次预览按需重新生成
- 建议映射方式：
  - key = `abs_dds_path + mtime_ns + size`
  - png 文件名使用 key 的 hash
  - 可避免“同名不同内容”脏缓存

## 6) 分阶段落地

- Phase A（快速收益）
  - `Music` 分类先接缓存
  - 接入 `invalidate_kind_cache("Music")`
- Phase B
  - 扩展到 `Stage`/`Map`/`Event`/`DDSImage`
- Phase C
  - 统一写操作后缓存失效钩子
  - 接入 `dds_preview` 失效删除

## 7) 风险与兜底

- 风险：缓存漏失效导致 UI 显示旧数据
- 兜底：
  - 保留“刷新”按钮强制重扫并重建缓存
  - 缓存版本号升级时自动清空
