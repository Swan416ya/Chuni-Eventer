# 启动耗时定位与 P0 修复建议

## 本次日志结论（你给的 102-128 行）

基于日志（`python -m chuni_eventer_desktop`）可直接分段：

- `+692ms -> +1308ms`：构造 `MainWindow` 前后，约 **0.6s**（可接受）
- `+1308ms -> +4450ms`：`MainWindow.__init__` 早期阶段，约 **3.1s**（次级慢点，需再细分）
- `+4451ms -> +21265ms`：`ensure_acus_layout`，约 **16.8s**（**主瓶颈**）
- `+21265ms -> +21617ms`：窗口 show 与进入事件循环，约 **0.35s**（可接受）
- `+21797ms`：索引缓存命中，后台索引未触发（说明这次慢并不是索引重建导致）

结论：当前这次 20s+ 启动，主要慢在 **启动同步执行 `ensure_acus_layout`**。

---

## 为什么 `ensure_acus_layout` 会这么慢

当前 `ensure_acus_layout(...)` 在启动主链路里做了这些事：

1. 创建 ACUS 目录结构（这个很快）
2. `_seed_acus_from_bundled`（通常也快）
3. `refresh_chara_works_sorts_with_game(...)`（这一步重）

而 `refresh_chara_works_sorts_with_game` 内部会调用：

- `resolve_game_chara_works_sort_path(...)`
- 里面又会 `enumerate_game_data_roots(...)`
- 该函数使用 `root.rglob("*")` 递归扫描游戏根目录，目录大时非常慢（机械盘/杀软实时扫描时更明显）
- 之后还会读取/解析多个 XML 并写回 `WorksSort.xml` / `WorksNameSort.xml`

也就是说：**你现在每次启动都在做一次“深度文件系统扫描 + XML 合并写回”**，这正是 16s~20s 的来源。

---

## P0 修复方案（按收益优先）

## 1) 把 `refresh_chara_works_sorts_with_game` 从启动主链路移走（最高优先）

- 现状：在 `ensure_acus_layout` 里同步执行
- 改法：
  - `ensure_acus_layout` 只保留“创建目录 + 必要 seed”
  - works sort 刷新改为后台任务（窗口 show 后触发）
  - 后台完成后发非阻塞提示（或静默）
- 预期收益：直接移除 16s 级阻塞，首屏可见时间大幅下降

## 2) 给 works sort 刷新加“变更检测”（跳过无变化场景）

- 增加缓存签名（例如）：
  - `game_root` 路径
  - A000 目标 sort 文件 mtime/size
  - `ACUS/charaWorks` 下 `CharaWorks.xml` 数量 + 最新 mtime
- 启动时若签名未变，直接跳过刷新
- 预期收益：避免“每次启动都重复重算”

## 3) 降级目录扫描策略（避免 `rglob("*")` 全盘递归）

- 针对已知结构优先定向探测：
  - `data/A000/charaWorks`
  - `A000/charaWorks`
  - `bin/option/A000` 等常见路径
- 仅当定向探测失败时，才做全量递归兜底
- 预期收益：IO 量级显著下降，尤其在大目录环境

## 4) 启动路径内禁止任何“非必须 XML 回写”

- 启动阶段只读，进入“可交互”后再异步写回
- 避免杀软/同步盘对写入动作放大时延

## 次级慢点（3.1s）如何继续定位

当前日志里 `mainwindow: __init__ enter -> loading config` 还有约 3.1s，建议再细分两刀：

1. 在 `MainWindow.__init__` 中把日志放到：
   - `super().__init__` 前
   - `super().__init__` 后
   - `setWindowTitle/resize` 后
2. 同时在 `ManagerWidget` 构造入口/出口打点

这样能快速判断是：

- `MSFluentWindow` 基类初始化慢
- 还是 UI 组件构造慢
- 还是终端输出时序造成的观测误差

---

## 建议的落地顺序（最短路径）

1. **立刻改**：`ensure_acus_layout` 去掉同步 `refresh_chara_works_sorts_with_game`
2. **立刻加**：后台刷新 works sort（与后台索引同策略，非阻塞）
3. **补充**：works sort 刷新签名缓存，默认无变化跳过
4. **最后**：继续细分 3.1s 次级慢点并处理

---

## 预期效果

按当前日志推算，若先移除 `ensure_acus_layout` 中的重刷新：

- 启动从 ~21s 理论可降到 ~4-6s 区间（具体取决于 `MainWindow` 早期 3.1s 的最终来源）
- 后续再优化次级慢点，可继续逼近 3s 级首屏
