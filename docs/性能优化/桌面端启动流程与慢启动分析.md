# 桌面端启动流程与慢启动分析

本文描述 **Chuni Eventer Desktop** 从进程启动到主界面可交互之间依次执行的工作，解释常见「启动很慢」的原因，并给出可落地的优化方向。

> 入口命令：`python -m chuni_eventer_desktop`（或打包后的 `ChuniEventer.exe`）  
> 核心代码：`chuni_eventer_desktop/__main__.py` → `app.py` → `ui/main_window.py`

---

## 1. 建议的衡量口径

| 标记 | 含义 | 理想目标（参考） |
|------|------|------------------|
| **T0** | 进程启动 | — |
| **T1** | 主窗口 `show()`，用户能看到窗口 | ≤ 2～3s |
| **T2** | 默认「歌曲」页可滚动/可搜索 | ≤ 3～5s |
| **T3** | 后台任务结束（游戏索引、外部工具安装等） | 不阻塞 T2 |

**体感慢**通常来自两类阻塞：

1. **主线程同步阻塞**：在 `MainWindow()` 或首轮 `reload()` 里做重 IO / 重计算，窗口迟迟不出现或出现后仍卡住。  
2. **模态对话框阻塞**：游戏索引首次重建、Lite 版首次下载外部工具时会弹出进度框，用户感觉「一直卡在启动」。

## 2. 启动流程总览

```mermaid
sequenceDiagram
    participant Proc as 进程
    participant App as app.main
    participant MW as MainWindow
    participant Loop as Qt 事件循环
    participant BG as 后台线程

    Proc->>App: __main__ / main()
    App->>App: DLL 路径 / 崩溃日志 / QApplication
    App->>MW: MainWindow() 同步构造
    Note over MW: 读配置、建 ACUS 目录、ManagerWidget、SettingsPage、导航
    App->>Loop: show() + exec()
    Loop->>MW: QTimer(0) _apply_category(Music)
    Note over MW: scan_music + 歌曲卡片 + 同步解码封面 DDS
    Loop->>MW: QTimer(0) _ensure_game_index_background
    alt 无游戏索引缓存
        MW->>BG: 扫描游戏目录 rebuild index
        BG-->>MW: 模态进度框（阻塞交互）
    end
    Loop->>MW: QTimer(0) _bootstrap_external_tools
    alt 打包版首次且缺 ffmpeg 等
        MW->>BG: 下载安装 .tools
    end
```

## 3. 分阶段：启动时具体做了什么

### 3.1 进程入口（`__main__.py`）

| 步骤 | 说明 | 耗时特征 |
|------|------|----------|
| quicktex 子进程分流 | 若 argv 含 `--chuni-quicktex-worker`，只跑 DDS worker，不启 GUI | 正常 GUI 启动跳过 |
| `ensure_pyinstaller_dll_search_path()` | 打包运行时修正 DLL 搜索路径 | 打包版固定开销 |
| `app.main()` | 进入 GUI 主流程 | — |

### 3.2 应用层初始化（`app.py::main`）

| 步骤 | 代码位置 | 说明 |
|------|----------|------|
| Windows AppUserModelID | `_set_windows_appusermodel_id` | 任务栏图标归属，毫秒级 |
| 崩溃日志 | `_setup_crash_logging` | 创建 `.cache/logs/`，启用 `faulthandler` |
| 可选弹窗调试日志 | `CHUNI_DIALOG_DEBUG=1` | 写 `dialog_debug.log` |
| `QApplication` | — | Qt 初始化 |
| 主题与图标 | `setTheme(AUTO)`、`setWindowIcon` | qfluentwidgets + 图标文件 |
| **`MainWindow()`** | 见下一节 | **通常是最重的同步阶段之一** |
| `show()` + 设置 HWND 图标 | Windows ICO | 毫秒级 |

### 3.3 主窗口同步构造（`MainWindow.__init__`）

构造期间 **全部在主线程完成**，用户此时还看不到窗口（`show()` 尚未调用）。

| 顺序 | 动作 | 涉及模块 | 说明 |
|------|------|----------|------|
| 1 | 读配置 | `AcusConfig.load()` | 读 `ACUS/.config.json` |
| 2 | 解析工具路径 | `apply_resolved_paths_to_config` | 若 `.tools/` 已有 exe 则写回配置 |
| 3 | 准备工作区 | `ensure_acus_layout` | 创建 `ACUS/` 子目录、首次 seed、`.cache/dds_preview` |
| 4 | 构建工作区 UI | `ManagerWidget(embedded=True)` | 表格、预览区、**MusicCardsView** 等 |
| 5 | 构建设置页 | `SettingsPage(...)` | **一次性创建** 关于 / 游戏数据 / 外部工具 / 存档 / 实验性 五个面板 |
| 6 | 左侧导航 | `navigationInterface.addItem` × N | Fluent 侧栏 |
| 7 | 注册延迟任务 | `QTimer.singleShot(0, …)` × 3 | 见 3.4 |

**关于 `ensure_acus_layout`（历史瓶颈，已优化）**

- **旧行为**：启动时同步调用 `refresh_chara_works_sorts_with_game`，内部可能 `rglob("*")` 扫描整个游戏目录并写 XML，单次可达 **10～20s**（见 `.cache/logs` 旧日志与 [启动耗时定位与P0修复建议.md](./启动耗时定位与P0修复建议.md)）。
- **当前行为**（`acus_workspace.py`）：只做目录创建 + seed + 缓存目录，**不再**在启动主链路扫描游戏目录。works sort 刷新改在**游戏索引后台扫描成功之后**执行（`main_window._GameIndexWorker.run`）。

### 3.4 `show()` 之后：首轮事件循环（仍可能阻塞主线程）

三个 `QTimer.singleShot(0, …)` 会在窗口显示后的**第一轮**事件循环执行：

#### A. 默认加载「歌曲」分类 — `_apply_category("Music", …)`

调用链：

```
_apply_category → ManagerWidget.set_kind("Music") → reload()
  → scan_music(acus_root)          # glob music/**/Music.xml + 解析 XML
  → _apply_filter → _rebuild_music_cards()
  → MusicCardsView.set_items → _rebuild_grid()
```

**歌曲页的关键慢点**：`_rebuild_grid` 对**每一首**有封面的曲目的 jacket DDS **同步**调用 `dds_to_pixmap`（`music_cards_view.py`）。  
即使已有 `.cache/dds_preview/*.png` 缓存，曲数多时仍要：

- 多次磁盘 stat / 读 PNG  
- 创建大量 `FlipMusicCard` 控件  
- 布局计算

曲数 × 封面解码是 **T2（列表可流畅操作）** 的主要瓶颈之一。

#### B. 游戏数据索引 — `_ensure_game_index_background`

| 条件 | 行为 |
|------|------|
| 未配置 `game_root` | 异步 Toast 提示，不扫描 |
| 已有 `.cache/game_data_index.json` 且可读 | **立即返回**，几乎无成本 |
| 无缓存 | 启动 **QThread** + **模态** `QProgressDialog`，执行 `rebuild_and_save_game_index` |

索引重建包含（`game_data_index.py`）：

- `enumerate_game_data_roots`：对游戏根目录 `rglob("*")` 找 `A???` 数据包  
- 扫描 music / stage / ddsImage / ddsMap / chara / nameplate / trophy 等  
- 写入 `.cache/game_data_index.json`  
- 成功后刷新 ACUS charaWorks 排序表（`_GameIndexWorker` 内，`prewarm_dds_preview=False`）

**注意**：启动路径里索引 worker **已关闭** DDS 全量预热（`prewarm_dds_preview=False`）；但**手动在设置里重建索引**时默认仍可能预热（`prewarm_dds_preview=True`）。

#### C. 外部工具 bootstrap — `_bootstrap_external_tools`（仅打包 `frozen`）

- 首次启动且 `.tools` 缺少 ffmpeg 等：后台下载 zip 并解压，带模态进度。  
- 源码 `python -m` 开发运行：**跳过**。  
- 「懒人包」已内置 ffmpeg：写 flag 后直接跳过。

### 3.5 Python  import 图（冷启动固定成本）

`main_window.py` **模块顶层** import 了十余个对话框与子系统（地图、角色、谱面下载、任务、Stage 等）。  
导入 `MainWindow` 时会连带加载 `ManagerWidget`，后者又顶层 import `CharaAddDialog` 等。

在本机 `.venv-build` 上粗测（仅供参考）：

| 阶段 | 约耗时 |
|------|--------|
| `import chuni_eventer_desktop.app`（含 MainWindow 链） | ~0.8s |
| `import ManagerWidget` 模块 | ~0.3s |
| `ManagerWidget()` 构造 | ~50ms |
| `ensure_acus_layout` | ~3ms |
| `scan_music`（8 首） | ~16ms |

**曲数、磁盘、杀软、是否 onefile 打包** 会显著改变上述数字；打包 onefile 还有解压到临时目录的额外秒级开销。

## 4. 为什么启动慢：按影响排序

| 优先级 | 原因 | 典型场景 | 相关代码 |
|--------|------|----------|----------|
| **P0** | 首次无游戏索引缓存，模态扫描游戏目录 | 新装 / 刚填 game_root / 删了 cache | `main_window._ensure_game_index_background` |
| **P0** | 歌曲页同步解码全部 jacket DDS + 建卡片 | ACUS 曲多、缓存未命中、quicktex 冷启动 | `music_cards_view._rebuild_grid` |
| **P1** | 主窗口顶层 import 过多模块 | 每次冷启动 Python 解释器 | `main_window.py` 顶层 import |
| **P1** | 启动即构造完整 `SettingsPage` 五 Tab | 即使用户从不打开设置 | `main_window` + `settings_page.py` |
| **P1** | 打包 onefile 解压 + DLL | 双击 exe 首启 | PyInstaller / `frozen_runtime` |
| **P2** | 打包版首次下载外部工具 | Lite 单 exe 无 ffmpeg | `external_tools_bootstrap` |
| **已缓解** | 启动同步 refresh works sort + rglob 游戏盘 | 旧版本 | 已移出 `ensure_acus_layout` |
| **已缓解** | 索引尾部全量 DDS 预热 | 旧版本启动索引 | worker 内 `prewarm_dds_preview=False` |

## 5. 优化建议（可直接对照改代码）

### P0 — 优先做，对体感提升最大

#### 5.1 歌曲封面异步 / 分批加载

**问题**：`_rebuild_grid` 在主线程为每首歌同步 `dds_to_pixmap`。  
**方向**：

- 先建无封面占位卡片，列表立刻可滚动；  
- 用 `QTimer` / 线程池按可见区域或分批解码 DDS；  
- 解码结果写入现有 `.cache/dds_preview`，与预览逻辑共用。

**改动点**：`ui/music_cards_view.py`（`_rebuild_grid`、可选新建 loader）

#### 5.2 游戏索引：无缓存时不阻塞首屏

**问题**：无缓存时模态进度框 + 大目录 `rglob` 仍让用户觉得「启动卡住」。  
**方向**：

- 窗口先可交互，索引在后台跑，状态栏/非模态提示「索引构建中」；  
- 依赖索引的下拉/浏览器在索引完成前显示「加载中」或仅用 ACUS 数据。

**改动点**：`ui/main_window.py`（`_ensure_game_index_background`、`_show_index_progress_dialog`）

#### 5.3 限制 `enumerate_game_data_roots` 的扫描范围

**问题**：`game_data_index.enumerate_game_data_roots` 对 `game_root.rglob("*")` 全树递归。  
**方向**：先探测已知路径（`data/A000`、`bin/option/A*` 等），失败再 fallback rglob（与 works sort 优化思路相同）。

**改动点**：`game_data_index.py`

### P1 — 降低每次冷启动固定成本

#### 5.4 对话框按需 import（lazy import）

**问题**：`main_window.py` 顶层 import 全部 `*AddDialog`、下载对话框等，启动即加载。  
**方向**：仅在 `_on_add` / 打开对应功能时再 `import`（`ManagerWidget` 内部分 dialog 已这样做，可推广到 `main_window`）。

**改动点**：`ui/main_window.py` 顶层 import 列表

#### 5.5 设置页懒加载

**问题**：`SettingsPage` 构造时创建五个子面板。  
**方向**：首次切换到某 Segmented 项再实例化对应 panel（或 `QStackedWidget` + factory）。

**改动点**：`ui/settings_page.py`

#### 5.6 ACUS 分类扫描结果缓存

**问题**：每次进入分类都 `glob` + 解析 XML（歌曲还会触发封面解码）。  
**方向**：见 [分类懒加载与缓存同步方案.md](./分类懒加载与缓存同步方案.md)（`.cache/acus_index/<kind>.json` + 目录 fingerprint）。

**改动点**：`acus_scan.py` + `manager_widget.reload`

### P2 — 打包与运维

- **onefolder 分发**或启动缓存，减轻 onefile 解压。  
- 外部工具 bootstrap 改为非模态 + 可取消。  
- 增加启动埋点（T0～T3 写 `.cache/logs/startup_profile.log`），便于对比优化前后。

## 6. 如何自己定位慢在哪一段

### 6.1 看现有日志

| 日志文件 | 内容 |
|----------|------|
| `.cache/logs/index_scan.log` | 游戏索引扫描起止、进度、是否命中缓存 |
| `.cache/logs/settings_save.log` | 配置写入 |
| `.cache/logs/python_crash.log` | 未捕获异常 |

若启动 ~20s 且 `index_scan.log` **没有** `scan_start`，慢点不在索引，而在 **MainWindow 构造** 或 **歌曲 reload**。

### 6.2 Python import 耗时

```powershell
cd "E:\Python Project\Chuni-Eventer"
.\.venv-build\Scripts\python.exe -X importtime -m chuni_eventer_desktop 2> importtime.log
```

查看 `importtime.log` 中 `main_window`、`manager_widget`、`qfluentwidgets` 等行的 cumulative time。

### 6.3 粗粒度手动打点（临时）

在 `app.py` / `MainWindow.__init__` / `ManagerWidget.reload` 入口加：

```python
import time
print(f"[startup] {time.perf_counter():.3f} label", flush=True)
```

对比各 label 间隔即可分段。

### 6.4 快速对照实验

| 实验 | 若明显变快，说明瓶颈在 |
|------|------------------------|
| 临时注释 `QTimer` 里 `_apply_category(Music)` | 歌曲 scan + 封面解码 |
| 清空 `game_root` 或保证索引 cache 存在 | 游戏目录扫描 |
| 源码运行 vs 打包 exe | 打包 / onefile 解压 |
| ACUS 内只有少量 music | 曲数与卡片/DDS 线性相关 |

## 8. 推荐落地顺序（实施优化时）

1. **歌曲封面异步加载** — 不改功能，只改加载时机，对「窗口出来仍卡几秒」最有效。  
2. **游戏索引改非模态后台** — 改善首次配置 game_root 的体验。  
3. **`main_window` 按需 import + 设置页懒加载** — 降低每次冷启动 ~0.5～1s+（视机器而定）。  
4. **ACUS 分类 JSON 缓存** — 二次进入分类、切换 Tab 更快。  
5. **游戏目录扫描定向路径** — 索引与 works sort 共用，缩 IO。

完成 P0 后，多数环境 T1 应落在 **2～4s**，T2 取决于曲库规模；P1/P2 继续压缩冷启动固定成本。
