# Chuni Eventer 启动性能优化方案

## 1. 目标与现状

- **现状问题**：从双击到可交互超过 20s，用户感知为“应用卡住”。
- **优化目标（建议）**：
  - 冷启动（首次打开）可见主窗口时间（TTFW） <= 3s
  - 冷启动到可完成基础操作（TTI） <= 5s
  - 首次重建索引等重任务转为后台，不阻塞主线程与首屏交互
- **衡量口径**：
  - `T0`：进程启动
  - `T1`：主窗口 `show()`
  - `T2`：主列表首次可滚动/搜索
  - `T3`：后台任务（索引、预热）完成

## 2. 当前启动阶段在做什么（基于现有代码）

### 2.1 入口与应用初始化

- 入口：`run_desktop.py` -> `chuni_eventer_desktop.app.main()`
- 应用初始化主要动作：
  - `ensure_pyinstaller_dll_search_path()`
  - 创建 `QApplication`
  - 设置主题、图标、Windows 任务栏属性
  - 实例化 `MainWindow`

### 2.2 主窗口构建（`MainWindow.__init__`）

- 读取配置：`AcusConfig.load()`
- 创建/校验工作区目录：`ensure_acus_layout(...)`
  - 包含 `refresh_chara_works_sorts_with_game(...)`（可能触发 XML 扫描/合并）
- 初始化 UI 结构（导航、搜索栏、管理区）
- `QTimer.singleShot(0, self._apply_category("Music", ...))`
  - 触发 `ManagerWidget` 首次数据加载（通常涉及 ACUS 扫描）
- `QTimer.singleShot(0, self._ensure_game_root_first_run)`
  - 如果配置了 `game_root` 且不存在缓存索引，则同步执行重建索引
  - 重建索引结束前，会显示模态进度框并持续占用 UI 线程

### 2.3 索引重建（主要重任务）

- `rebuild_and_save_game_index(...)` 包含：
  - 递归枚举游戏数据包 `A???` 目录
  - 扫描 music/stage/ddsImage/ddsMap/chara/nameplate/trophy
  - 写入 `.cache/game_data_index.json`
  - **额外执行** `prewarm_game_dds_preview_pngs(...)`
    - 扫描大量 `ddsMap/**/*.dds`
    - 执行 DDS -> PNG 转换（CPU + IO 重）

> 结论：当前启动路径把“可延后的大任务”放在了首轮 UI 生命周期里，且包含大量文件系统遍历与图像转换，导致首启和冷启动明显变慢。

## 3. 启动阶段职责拆分（建议）

把启动分成三层，只保留最小必需动作在首屏路径：

- **L0（必须同步，<500ms）**
  - 进程与 Qt 基础初始化
  - 读取轻量配置
  - 构建壳层 UI（窗口、导航、空列表骨架）
- **L1（可交互前必要，<2s）**
  - 仅加载当前 Tab 所需的最小数据（例如 ACUS music 基础列表）
  - 使用缓存，不做全量重扫
- **L2（后台异步，不阻塞）**
  - 游戏索引重建
  - dds 预览预热
  - works sort 补全、复杂校验

## 4. 优化方案（按优先级）

## P0（应先做，通常可直接把 20s 降到 3~6s）

- **方案 A：把索引重建从“启动同步”改为“后台任务”**
  - 现状：`_ensure_game_root_first_run()` 在启动即同步执行重建
  - 调整：
    - 启动只判断缓存是否可用；不可用时先进入主界面
    - 在状态栏/Toast 提示“正在后台建立游戏索引”
    - 完成后再刷新依赖索引的下拉与浏览器
  - 预期收益：避免启动阶段被大规模扫描阻塞

- **方案 B：`prewarm_game_dds_preview_pngs` 脱离索引流程**
  - 现状：索引重建尾部强制执行预热
  - 调整：
    - 默认不在启动期间预热
    - 改为“按需生成 + 空闲时增量预热”
  - 预期收益：显著减少首次启动 CPU/IO 峰值

- **方案 C：为首屏列表提供“快速路径”**
  - 启动仅加载当前分类最小字段（id/name），其他字段惰性读取
  - Manager 首次渲染显示 skeleton，避免“空白等待”
  - 预期收益：提升“已启动”的主观体验

## P1（中优先级，进一步优化冷启动）

- **方案 D：减少模块级重导入与启动时大 import**
  - `main_window.py` 顶层导入了大量 dialog/功能模块
  - 调整为“按需导入”（点击某功能时再 import）
  - 预期收益：降低 Python 冷启动与解释器初始化耗时

- **方案 E：`ensure_acus_layout` 轻量化**
  - 启动只做目录存在性检查；复杂 XML 合并放后台
  - 对 `refresh_chara_works_sorts_with_game` 做变更检测（mtime/hash）再执行
  - 预期收益：降低每次启动的固定成本

- **方案 F：缓存策略升级**
  - 缓存写入增加版本 + 源目录签名（mtime 快照）
  - 签名未变时直接复用缓存，跳过全量扫描
  - 预期收益：二次启动速度稳定

## P2（长期项）

- **方案 G：并行扫描与任务分片**
  - 多目录扫描可并行化（线程池/进程池，注意 GIL 与 IO 型任务）
  - 大 XML 解析与图像转换任务分批调度
- **方案 H：打包层优化**
  - 针对 PyInstaller onefile 的解包启动开销，评估 onefolder 分发或启动缓存策略
  - 清理不必要依赖，减少 import graph

## 5. 推荐落地顺序（两周版本）

- **第 1 阶段（1-2 天）**
  - 加启动埋点（T0/T1/T2/T3）
  - 将索引重建改后台执行
  - 将 dds 预热从索引链路中移除
- **第 2 阶段（2-3 天）**
  - 首屏骨架 + 最小数据加载
  - `main_window.py` 顶层导入按需化
- **第 3 阶段（2-3 天）**
  - 缓存签名与增量失效
  - `ensure_acus_layout` 重任务拆分

## 6. 风险与回滚策略

- **一致性风险**：后台索引未完成时，部分下拉数据不完整
  - 处理：UI 显示“索引构建中”，并提供手动刷新按钮
- **线程安全风险**：后台任务完成后更新 UI 必须回到主线程
  - 处理：统一通过 Qt signal/slot 回传结果
- **兼容风险**：不同用户目录结构差异大
  - 处理：保留“手动重建索引”入口，失败时给出可读错误
- **回滚策略**：
  - 每个优化点保留开关（配置项/环境变量），异常可快速切回旧行为

## 7. 下一步实现建议（可直接进入开发）

- 先做 P0 三项（后台索引、移除启动预热、首屏快速路径），这是最可能立刻把 20s+ 启动体感降到可接受范围的组合。
- 完成后基于埋点数据再决定 P1 的优先项（通常按需导入 + 缓存签名收益最高）。
